# Servers & Groups

## Overview
Servers are the primary community spaces in Chip — analogous to Discord servers. They are created and owned by `super_owner` or `investor`-role users. Each Server hosts Pages, Groups, Features, and its own member community.

---

## Server Types

| Type | Created by | Limit | Monthly cost |
|---|---|---|---|
| `owner` | `super_owner` (seeded at install — the Super Organisation) | Unlimited | None |
| `standard` | `investor` | One per user | Sum of Features subscribed to that Server |

There is exactly **one** `owner`-type Server (Limbo). It is the Super Organisation and cannot be created via the UI.

---

## Server Data Model

```
servers
  id                ulid
  name              string (unique)
  slug              string (unique, URL-safe)
  description       text nullable
  type              enum: owner | standard
  owner_id          FK → users
  join_mode         enum: link | invite_list | manual   (controls who can join — see Membership)
  member_cap        integer  (base from Investor payment; increases with Member Slot add-on)
  group_cap         integer  (base from boost tier)
  boost_tier        integer  default 0
  credit_balance    integer  (pence; accumulated boost payments not yet invested)
  preview_duration  integer  (hours; how long a new visitor's preview window lasts)
  created_at / updated_at
```

## Membership

### Join Mode Configuration

The Server owner sets a `join_mode` on their Server. This controls the **self-service** join path available to users. The server owner can **always** add any existing platform user directly regardless of `join_mode` — including users who have been previously denied a preview extension or demo access (see below).

| Mode | Who can join | How |
|---|---|---|
| `link` | Anyone who clicks an invite link | Link is generated by owner/moderator; recipient verifies identity via OTP; no prior account required |
| `invite_list` | Only people whose email is on the Server's invite list | Two paths: (1) person clicks a link → OTP verifies their email → if email is on the list they are added; (2) when a user registers or logs in, the system automatically checks all `invite_list` servers — if their email is on the list they are silently added as a member without needing any link |
| `manual` | Only existing platform users added by owner/moderator | Owner or moderator searches by email; the person must already have a Chip account; they are added directly — no link or OTP |

#### Invite List Data Model

Used when `join_mode = invite_list`. The owner or moderator pre-populates email addresses.

```
server_invite_list
  id          ulid
  server_id   FK → servers
  email       string             (plain; matched case-insensitively against user accounts)
  added_by    FK → users
  added_at    timestamp
```

Unique constraint on `(server_id, email)`. When a user registers or logs in, a background job checks if their email appears in any `server_invite_list` and auto-enrols them on matching servers.

#### OTP on Link Joins

When `join_mode = link` or `invite_list` (link path): the recipient opens the invite link, enters their email or phone, receives a numeric OTP, and verifies it before the join is recorded. No anonymous joins. See `blueprint/content-visibility.md`.

### Time-Limited Preview

- When a registered user first visits a Server (before joining), they receive a **time-limited preview window**.
- Duration is set by the Server owner via `preview_duration`. Default TBD, configured by `super_owner`.
- During the preview, the user can see Pages the Server owner marks as preview-accessible (`preview_visible = true` on the page).
- After expiry, all Server content is hidden until they formally join.
- A user can **request a 1-week extension once per week**.
  - The request is reviewed by the Server owner or a moderator (`server.members.preview_extend` permission).
  - The reviewer can **approve** or **deny** the request, with an optional message that is shown to the requesting user.
  - If **denied**, the user is permanently blocked from submitting further **self-service** extension requests for that Server.
  - **Owner override**: the server owner can always directly extend a user's preview (via `server.manage`) or directly add the user as a full member regardless of any prior denial. The denial only prevents the user from placing new self-service requests.

#### Preview Extension Request Data Model

```
server_preview_extension_requests
  id              ulid
  server_id       FK → servers
  user_id         FK → users
  status          enum: pending | approved | denied
  message         string nullable    (optional message from reviewer, shown to user)
  requested_at    timestamp
  resolved_at     timestamp nullable
  resolved_by     FK → users nullable
```

Unique constraint on `(server_id, user_id)` where `status = pending` — only one pending request at a time. If `status = denied`, no further **self-service request row** may be inserted for that `(server_id, user_id)` pair. Owner-initiated actions (direct member add, direct preview extension) bypass this table entirely.

### Demo Feature Access Requests

**Auto-demo access (most features):** When a feature has `demo_accessible = true` (the default, set by `super_owner`), any user in `preview` status on the Server automatically gets limited access to that feature — no request or approval needed. Usage is capped at `demo_usage_limits` defined on the feature. This is the standard path.

**Manual request flow (opt-out features only):** When a feature has `demo_accessible = false`, users must request access manually:

- Any user in a Server preview or active member can **request access to a specific demo feature** on that Server.
- The Server owner reviews and **approves** or **denies** the request, with an optional message shown to the requesting user.
- If **denied**, the user is permanently blocked from placing further **self-service** demo access requests for that feature on that Server.
- **Owner override**: the server owner can always grant demo feature access directly via `server_feature_access` (using `server.access.configure` permission) regardless of any prior denial.

#### Demo Feature Access Request Data Model

```
server_demo_access_requests
  id              ulid
  server_id       FK → servers
  user_id         FK → users
  feature_id      FK → features
  status          enum: pending | approved | denied
  message         string nullable    (optional message from server_owner, shown to user)
  requested_at    timestamp
  resolved_at     timestamp nullable
  resolved_by     FK → users nullable
```

Unique constraint on `(server_id, user_id, feature_id)` where `status = pending`. If `status = denied`, no further **self-service request row** may be inserted for that `(server_id, user_id, feature_id)` combination. Owner-initiated direct grants via `server_feature_access` bypass this table entirely.

### Member Data Model

```
server_members
  id                  ulid
  server_id           FK → servers
  user_id             FK → users
  server_role         enum: server_owner | moderator | supporter | member
  joined_at           timestamp nullable    (null while in preview)
  status              enum: preview | active | suspended | banned
  preview_expires_at  timestamp nullable
  suspended_until     timestamp nullable    (null unless status = suspended; system auto-lifts when past)
  suspension_reason   string nullable       (optional moderator note)
  ban_reason          string nullable       (optional; recorded when status set to banned)
  created_at / updated_at
```

Unique constraint on `(server_id, user_id)`.

> **Status rules:**
> - `preview` — user is viewing under a time-limited preview, not yet a formal member.
> - `active` — full member; normal access.
> - `suspended` — temporarily inactive; all server access revoked until `suspended_until` expires or is manually lifted. Row retained. Counts against member cap.
> - `banned` — permanently removed. Row retained to block future join attempts. Does **not** count against member cap. Only `server_owner` can lift a ban.
> - **Kicked** and **leave** are actions — not statuses. Both result in the `server_members` row being **deleted**. The event is recorded in `access_logs`. A kicked/left user can rejoin normally (subject to `join_mode`). To prevent someone from rejoining, use `banned`.

### Server Roles

| Role | Granted by | Requirement |
|---|---|---|
| `server_owner` | System on Server creation | Must hold `investor` or `super_owner` platform role |
| `moderator` | `server_owner` manually | Must currently hold `supporter` role on this Server |
| `supporter` | System on boost payment or subscription | Has boosted or subscribed to this Server |
| `member` | System on join | Registered user |

A user holds exactly one Server role per Server — the highest applicable role.

**Moderator → supporter dependency:** Moderator status is only valid while `supporter` status is maintained. If a user's `supporter` status is lost (e.g. boost refunded, subscription cancelled and expired), they are automatically demoted to `member` — not `supporter` — and the `moderator` role is removed. This demotion is handled as a background job triggered by payment refund / subscription expiry events.

**Suspension and ban:** A suspended member has `status = suspended`; access is revoked until `suspended_until` or manual lift. A banned member has `status = banned`; they cannot rejoin and are excluded from member cap counts. Only `server_owner` can lift bans. Kicking and leaving delete the row; those users can rejoin (subject to `join_mode`).

## Boost & Credit Pool

### How Boosting Works

1. Any registered user visits a Server and chooses to boost it.
2. They pay a one-off amount — between £0 and the maximum the Server owner has set for that feature context.
3. Payment enters the Server's `credit_balance`.
4. The user gains `supporter` server role on that Server (if not already moderator or owner).
5. Server owner grants feature/page access to supporters at their own discretion.

### Boost Tiers

The Server's `boost_tier` rises as the cumulative all-time boost income crosses thresholds defined by the `super_owner` in platform config. Higher tiers unlock increased limits.

| Tier | Example unlocks (values set by super_owner) |
|---|---|
| 0 | Base member cap, base group cap, base feature usage limits |
| 1 | +N member cap, higher feature usage limits |
| 2 | More groups, higher limits, additional privileges |
| … | Defined in platform config |

### Investing Credits

Server owner allocates credits from `credit_balance` toward:
- **Features** — subscribe a new Feature or renew an existing one on the Server
- **Member Slots** — purchase `member_cap` increments
- **Monthly bill offset** — applied as credit against the next monthly feature subscription invoice

Unspent credits automatically offset the next monthly bill.

### Credit Pool Rules

- Credits are locked to the Server — no cash out.
- **Refund policy on boosts and donations**: 80% of the original payment refunded if requested within 7 calendar days; no refund after 7 days. Credits already invested from the pool before a refund is requested cannot be reversed. See `blueprint/payments-subscriptions.md` for full details.
- Credit allocation is recorded in `server_credit_transactions`.

```
server_credit_transactions
  id              ulid
  server_id       FK → servers
  type            enum: boost_in | donation_in | feature_invest | slot_invest | bill_offset
  amount_pence    integer
  description     string nullable
  created_by      FK → users nullable
  created_at
```

---

## Member Slots (N)

- A base value `member_cap` is set when the Investor creates their Server (included in the Investor one-off payment, value TBD).
- Additional slots are a separately subscribable add-on (priced per increment, e.g. per 10 extra members).
- Any Server member can boost the Server to contribute to the credit pool; the Server owner can invest those credits in additional slots.

## Groups

### What is a Group?
A Group is a named sub-group within a Server. Groups allow the Server owner to organise members and scope Page visibility and Feature access.

### Group Management
- Created and managed by the Server owner or a moderator (with permission granted by Server owner).
- Any active Server member can be added to a Group.
- A member can belong to multiple Groups in the same Server.

### Group Data Model

```
groups
  id              ulid
  server_id       FK → servers
  name            string (unique within server)
  description     text nullable
  created_by      FK → users
  created_at / updated_at
```

### Group Membership Table (`group_members`)

```
group_members
  id              ulid
  group_id        FK → groups
  user_id         FK → users
  added_by        FK → users
  created_at
```

Unique constraint on `(group_id, user_id)`.
